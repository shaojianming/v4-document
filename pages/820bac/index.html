# Integration process

# Authorization Process

Here is the list of APIs that need to be integrated：

authorizationPrepare
authorizationApplyToken


  1. Call the authentication API to fetch the wallet authentication link. Once the user completes the authentication by accessing the authentication link, they will be redirected to the specified redirect address. The merchant can obtain the authorization code "code" from the redirect address.

  2. To call the authorization interface, pass the code obtained in the first step to acquire the payment token.

# Payment Process


Below is the list of APIs that need to be integrated:

Accept a Payment

Call the Accept a Payment with the token and bizType as parameters.

# Key input parameters:
Parameter field Parameter type Reduired description
tokenDetail.token String M Pass the token obtained from the Authorization API as an existing field
bizType String M For an existing field, add a new type identifier for authentication authorization and pass the fixed value of "CodeGrant"
merchantUserId String M Cardholder ID for merchant website

When calling the place order and payment API, it is mandatory to include the parameter "merchantUserId" in the input. It must be consistent with the "merchantUserId" provided during authentication. Otherwise, an error will occur stating that the token does not exist.

# Cancel authorization

When initiating a cancellation of authorization, there are two possible scenarios:

  • Call the API to cancel authorization from the merchant's side;
  • Wallet users can cancel authorization directly within the wallet platform.

Here is the list of APIs that need to be integrated：

Cancel Authorization API

If you need to cancel authorization，you can call the Cancel Authorization API to initiate the cancellation，Once the authorization is canceled, the token will become invalid.

# Asynchronous notification

The following three scenarios can trigger asynchronous notifications：

  1. Successful user authorization；
  2. Merchant-initiated authorization cancellation；
  3. User-initiated authorization cancellation on the wallet side.


Parameter required attribute description:Mandatory (M), optional (O), conditional (C) POST JSON
Parameter field Parameter type Reduired description
token String C Revoked authorization token
notifyType String C Possible values：
  • ACCESS_TOKEN_CREATION
  • ACCESS_TOKEN_CANCEL_OF_MERCHANT
  • ACCESS_TOKEN_CANCEL_OF_MERCHANT_USER

When the HTTP code is 200, it indicates that the merchant has received the message, and no additional information needs to be returned by the merchant； The response message will be retried twice in case of an abnormality
